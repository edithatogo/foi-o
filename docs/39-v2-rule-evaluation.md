# V2 deterministic rule evaluation

FOI-O already contains a deterministic `oia_rules` implementation and routes
process/CLI clock calculations through it. V2 therefore treats rule/process
coupling as an **implemented exemplar to validate empirically**, not as a future
architecture to invent.

## Evaluation question

Given independently curated inputs, the law and calendar applicable at event
time, and an explicit jurisdiction profile, does the rule module return the
expected candidate result and a complete, reproducible trace without making a
legal compliance finding?

## Independent oracle

Evaluation fixtures must be derived independently from the implementation under
test. Acceptable sources include human-checked statutory examples, official
guidance, Ombudsman or review findings, and adjudicated public-request cases.
Outputs generated by `oia_rules`, or by a language model without independent
review, are not a gold oracle.

The repo-local candidate set at
`tests/fixtures/oia_rules/oia-event-time-independent-candidates.json` is a
pre-review evaluation input, not a gold oracle. It pins the SHA-256 of the
approved authoring fixture and the test suite enforces disjoint case identifiers
and input dates. Its synthetic expected outputs may be used to detect runtime
drift, but `humanReview.promotionAllowed` remains false until a human reviewer
independently recalculates and approves every case.

## Required evaluation families

The test set should cover ordinary working-day calculations, weekend and public
holiday boundaries, Christmas/New Year exclusions or other special calendar
rules, leap years, supported transfer/extension cases, missing or conflicting
inputs, historical law/calendar versions, and cases where the clock start is not
observable from the public record.

## Outputs and metrics

Every run should preserve the rule and parameter versions, source references,
input epistemic statuses, calculation steps, warnings, derivability, and any
abstention reason. Report exact-date agreement, trace completeness, source and
calendar version resolution, abstention correctness, unsafe compliance-inference
rate, and a reviewed divergence taxonomy.

A disagreement is not automatically an implementation defect: it may reflect a
fixture error, source-version error, missing input, ambiguous law, or public-data
non-observability. All divergences remain inspectable.

## PIC relationship

PIC parameters, fixtures, and traces may be produced as candidate consumer
evidence from this work. PIC does not replace the FOI-O runtime. Candidate
fixtures or mappings require human promotion and retain their source and
independence records.

## Certification boundary

The automated output is a candidate deadline or process result. Determining
whether an authority legally complied, whether an extension was valid, or which
receipt date controls remains a human or official-source conclusion.
